## 一、角色分工

### 1. Task 指派者
- ✅ 分析專案現狀與技術架構
- ✅ 決定任務順序與優先級
- ✅ 撰寫/改進 Spec 草稿
- ✅ 將任務轉譯為 `task_prompt.md`
- ✅ 維護總覽與排程文件
- ❌ 不直接寫實作程式碼

### Task 指派者應閱讀
- `README.md`
- `sync_summary.md`
- `planning/project_goals.md`
- `planning/mvp_todo.md`
- `planning/tech_spec_notes.md`
- `planning/implementation_order.md`
- `planning/coding_habits.md`
- `specs/input_keymap.md`
- `task_backlog.md`
- 對應任務的 `reviews/*.md`

### 2. 實作者
- 🔧 讀取 `task_prompt.md` 獲取當前任務
- 🔧 依任務內列出的參考檔案實作
- 🔧 完成後更新 Prompt 底部「任務狀態」
- 🔧 回報實作摘要與遇到的問題
- ❌ 不修改 Prompt 的任務內容

### 實作者應閱讀
- `README.md`
- `task_prompt.md`
- `specs/input_keymap.md`
- `task_prompt.md` 內列出的參考檔案

### 3. 驗收者
- ✅ 檢查實作結果（功能/架構/風格）
- ✅ 對照 acceptance criteria
- ✅ 建立 `reviews/*.md`
- ✅ 決定通過、條件通過或退回
- ✅ 更新進度追蹤文件

### 驗收者應閱讀
- `README.md`
- `task_prompt.md`
- 實作者修改的檔案
- `code_review_template.md`
- 對應任務的歷史 `reviews/*.md`
- `sync_summary.md`

## 三、任務流程

```
┌─────────────────────────────────────────────────────────────┐
│  Step 1: Task 指派者 分析現狀                                 │
│  - 檢查專案結構                                              │
│  - 確認當前進度                                              │
│  - 識別下一步目標                                            │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 2: Task 指派者 撰寫 Spec                               │
│  - 定義任務目標（Why）                                       │
│  - 明確驗收標準（Acceptance Criteria）                       │
│  - 提供技術約束與參考檔案                                    │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 3: Task 指派者 產出 Prompt                             │
│  - 覆寫 `task_prompt.md`                                      │
│  - 包含完整背景與任務描述                                    │
│  - 任務狀態設為「進行中」                                    │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 4: 實作者 執行任務                                      │
│  - 讀取 Prompt 檔案                                          │
│  - 實作程式碼                                                │
│  - 完成後更新任務狀態為「已完成」                            │
│  - 填寫實作摘要、修改檔案、備註                              │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 5: 驗收者 驗收                                          │
│  - 檢查修改的檔案                                            │
│  - 對照驗收標準                                              │
│  - Code Review（架構/風格/邊界）                             │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
        ┌──────────────────┴──────────────────┐
        ▼                                     ▼
┌───────────────┐                   ┌─────────────────┐
│  驗收通過      │                   │  需要修正        │
│  → 進入下一任務 │                   │  → 回饋給實作者  │
│  → 建立 review  │                   │  → 更新 Prompt   │
│  → 更新進度    │                   │  → 重新實作      │
└───────────────┘                   └─────────────────┘
```

## 四、Prompt 模板結構

每個 `task_prompt.md` 必須包含以下章節：

### 1. 任務背景
- 專案概覽
- 已完成基礎
- 當前目標

### 2. 當前任務
- 任務標題
- 任務描述（Why + What）
- 具體需求（How）
- 驗收標準（Checklist）

### 3. 技術約束
- 必須使用的現有 API
- 禁止做的事
- 命名/風格規範

### 4. 參考檔案
- 需要讀取的關鍵檔案路徑

### 5. 輸出要求
- 完成後如何標記
- 需要回報的資訊

### 6. 任務狀態
- 進行中 / 已完成 Checkbox
- 實作摘要
- 修改檔案清單
- 備註/問題

## 六、溝通原則

1. **單一任務原則** - 一個 Prompt 只包含一個可完成的任務
2. **角色邊界清楚** - 指派者不寫實作，實作者不改規格，驗收者不補寫功能
3. **文件即契約** - Prompt 即規格，驗收即標準
4. **自我記錄** - 實作者更新任務狀態，驗收者建立 review，指派者更新總覽

- [ ] 開始協作！
