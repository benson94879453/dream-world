# Kimi × Codex 協作工作流程

## 角色分工

### Kimi (規劃與驗收)
- ✅ 分析專案現狀與技術架構
- ✅ 撰寫/改進 Spec 草稿
- ✅ 將任務轉譯為 Codex Prompt
- ✅ 驗收實作結果
- ✅ 判斷是否符合預期
- ✅ 決定下一步任務

### Codex (實作)
- 🔧 讀取 `codex_prompt.md` 獲取當前任務
- 🔧 執行具體程式碼實作
- 🔧 完成後更新任務狀態
- 🔧 回報實作摘要與問題

## Prompt 模板結構

每個 `codex_prompt.md` 應包含：

### 1. 任務背景
- 專案概覽 (給新進上下文)
- 已完成基礎 (Phase 現況)
- 相關系統現狀

### 2. 當前任務
- 任務標題
- 任務描述 (Why + What)
- 具體需求 (How)
- 驗收標準 (Checklist)

### 3. 技術約束
- 必須使用的現有 API
- 禁止做的事
- 命名/風格規範

### 4. 參考檔案
- 需要讀取的關鍵檔案路徑

### 5. 輸出要求
- 完成後如何標記
- 需要回報的資訊

### 6. 任務狀態
- 進行中 / 已完成 Checkbox
- 實作摘要 (Codex 填寫)
- 修改檔案清單 (Codex 填寫)
- 備註/問題 (Codex 填寫)

## 驗收標準範例

### 功能驗收
- [ ] 功能按預期運作
- [ ] 無 Console 錯誤
- [ ] 邊界情況處理 (null check, 異常輸入)

### 架構驗收
- [ ] 使用現有 API，無重複造輪子
- [ ] 資料邊界正確 (Resource vs Instance)
- [ ] 不違反 EventBus 規範

### 風格驗收
- [ ] 命名符合專案慣例
- [ ] 程式碼結構清晰
- [ ] 適當的註解

---

## 溝通原則

1. **Kimi 不直接寫實作程式碼** - 只寫 Spec、Prompt、驗收
2. **Codex 不修改 Spec** - 有疑問在「備註/問題」區提出
3. **單一任務原則** - 一個 Prompt 只包含一個可完成的任務
4. **快速迭代** - 小步快跑，及時驗收
